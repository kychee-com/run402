## Why

Gateway v1.85 / v1.87 (`016b7f8c`, `2b2055f4`) introduced a passkey-fresh **operator approval** for high-stakes writes — minted as a short-lived token carried in an `X-Run402-Write-Auth` header that the gateway now requires *in addition to* the control-plane session bearer before a **wallet-less human operator** may provision or deploy. The dual-header gate (`requireWriteAuthCovers`) protects `provision` (`org.project.create`) and `deploy`/`apply` (`project.deploy`): a control-plane session alone "stays read-only on these routes" and returns `403 WRITE_AUTH_REQUIRED`.

The public CLI mints the control-plane session (`run402 operator login --loopback`) but has **zero approval-token plumbing** — no ceremony to mint it, no cache, no header. Worse, the default `NodeCredentialsProvider.getAuth` only ever returns wallet (SIWX) headers; it never falls back to the cached control-plane session. So today a wallet-less human can `operator login` and read their overview, but every write fails. This change closes the wallet-less write path. (The SIWX wallet path that agents use is unaffected — `walletAuthOrWriteAuthProvision` still accepts wallet auth with no approval token.)

**Framing (per an external design review — GPT-5.5 Pro; transcript in `run402-private/docs/consultations/write-auth-loopback-design-review.md`):** the public product concept is **operator approval** — a passkey-signed human authorization for a mutating capability. `X-Run402-Write-Auth` / `WRITE_AUTH_REQUIRED` stay buried as the SDK/gateway transport. The client surface (verb, SDK seam, error) speaks "approval," and — critically — the human approval is **never ambient**: it must not leak into agent (MCP) tool calls.

> **Scope note:** the large item from the upgrade review. It spans `core` (token cache), the SDK credential-resolution + kernel header path, the operator ceremony, and the CLI. It ships as **two PRs under this one OpenSpec change** — auth plumbing first, then the approval ceremony — keeping the product concept (wallet-less operator writes) singular while lowering rollout risk.

## What Changes

- **NEW — `run402 operator approve` (the public verb).** Mints a passkey-fresh operator approval for provision/deploy via a headless loopback-PKCE ceremony. `run402 operator write-auth` ships as a hidden expert alias. Isomorphic SDK seams `r.operator.approval.requestChallenge({ cliRedirectUri, codeChallenge, state })` → `POST /agent/v1/control-plane/write-auth/challenges`, and `r.operator.approval.exchangeClaimCode({ code, codeVerifier, redirectUri, state })` → `POST /agent/v1/control-plane/write-auth/cli/token` (the `/write-auth/` paths stay as transport). Mirrors the existing `buildCliAuthorizeUrl` / `exchangeCliToken` seams; hardened (redirect_uri in the exchange, `confirm_url` validation, single-use code, never prints the token).
- **NEW — surface-aware, deterministic credential resolution (NOT ambient).** `NodeCredentialsProvider` takes `{ surface: "cli" | "mcp" | "sdk", authMode?: "auto" | "wallet" | "operator" | "none" }`. Resolution selects **exactly one** credential class with **no silent fallback** between classes. CLI defaults to `auto` (wallet → else cached operator approval). **MCP defaults to `wallet` only** — it never silently spends the human's cached approval; a write that needs approval surfaces a structured `OperatorApprovalRequiredError` for the agent to relay. SDK Node default resolves to `auto` only when explicitly constructed that way.
- **NEW — capability-driven dual-header attachment (NOT blanket).** `getAuth(path, metadata?)` receives `{ method, mutates?, capability? }` from the typed SDK method. The control-plane bearer + `X-Run402-Write-Auth` attach only when resolution selects the operator-approval class for a **mutating/gated capability** — never blanket on every cp-session request, never via a brittle client path allowlist. (This reverses the earlier Decision 2.)
- **NEW — approval token cache, bound to its session + origin + target.** `core/src/write-auth-session.ts` keeps the proven 0600/atomic discipline but adds `control_plane_session_hash`, `control_plane_principal_id`, `api_origin`, `scopes`, `org_id?`, `project_id?`, `minted_at` to the token shape. `loadLive…` returns `null` on cp-session-hash mismatch, principal mismatch, api-origin mismatch, scope/target mismatch, or expiry. The approval **dies with the cp-session** and is invalidated on every cp-session lifecycle change (login, logout, step-up/rotation), not just logout.
- **NEW — small kernel hardening (no endpoint-specific logic).** Make the provider-header merge **case-insensitive** and **credential-family-atomic**: if a request already set an auth header (`Authorization` / `SIGN-IN-WITH-X` / `X-Run402-Write-Auth`, any casing), the kernel does not merge a provider auth header over or beside it. Reframes the earlier "no kernel change" claim to "no endpoint-specific kernel logic."
- **NEW — typed `OperatorApprovalRequiredError`** (gateway code stays `WRITE_AUTH_REQUIRED`). Carries `{ code, principal: "operator", capability, next_actions[] }`; the structured remediation points at `run402 operator approve`.
- **NEW — `run402 operator status`** — shows operator-login + approval state, scopes, expiry, and org/project target (human-readable + JSON), making the dual-credential model legible to humans and agents.
- **NEW — interactive auto-approve-and-retry (TTY only).** `run402 deploy apply` / `provision`, when approval is missing AND stderr is a TTY, prompt to open the browser, run the ceremony, and retry once. **Never** in MCP, CI, or non-TTY — those return the structured error.
- **DOC FIX — control-plane session is no longer SIWX-equivalent for writes.** Correct the stale `core/control-plane-session.ts` comment and document the operator-approval model + `WRITE_AUTH_*` codes in `cli/llms-cli.txt` / `sdk/llms-sdk.txt`.

## Impact

- **core (`core/src/`):** new `write-auth-session.ts` (approval cache with session/origin/target binding; 0600/atomic discipline from `control-plane-session.ts`).
- **SDK (`sdk/src/`):** `namespaces/operator.ts` — new `Approval` sub-class (`r.operator.approval`); `node/credentials.ts` — `{ surface, authMode }` options + deterministic resolution + capability-metadata-driven dual-header attachment with cp-session fallback (CLI) / no-ambient (MCP); `kernel.ts` — case-insensitive + credential-family-atomic header merge, and thread per-request `mutates`/`capability` metadata into `getAuth`; `credentials.ts` — extend the `getAuth` signature to `getAuth(path, metadata?)`; `errors.ts` — `OperatorApprovalRequiredError` + `403 WRITE_AUTH_REQUIRED` mapping with `next_actions`.
- **CLI (`cli/lib/`):** `operator.mjs` — `approve` verb (+ hidden `write-auth` alias), `status` verb, lifecycle invalidation on login/logout/step-up; `provision`/`deploy` — TTY-only auto-approve-and-retry; the dual header rides via resolution.
- **MCP:** no ceremony tool; the SDK is constructed `surface: "mcp"` so writes needing approval return `OperatorApprovalRequiredError` (no ambient human authority). Document that an MCP host establishes approval out-of-band via the CLI.
- **Tests:** SDK unit (resolution matrix per surface/authMode; capability-driven attachment; kernel header-merge hardening; `403 → OperatorApprovalRequiredError`); core unit (cache round-trip + binding mismatches + expiry); CLI e2e (ceremony, `status`, TTY auto-approve, MCP-surface no-ambient) registered in the `package.json` allow-list; `sync.test.ts` SURFACE for the new verbs.
- **Docs:** `cli/llms-cli.txt`, `sdk/llms-sdk.txt` — the operator-approval model, `operator approve` / `operator status`, the surface/authMode semantics, and `WRITE_AUTH_*` codes; fix the stale `control-plane-session.ts` comment.
- **Cross-repo:** gateway shipped (`016b7f8c`, `2b2055f4`, run402-private). **Live-gateway verification is the blocking task 0.1** — expanded to cover origin isolation, scope/target binding, idle-TTL behavior on non-gated routes, and principal-selection safety (see `tasks.md`).
