## Why

Gateway v1.85 / v1.87 (`016b7f8c`, `2b2055f4`) introduced a passkey-fresh **write-auth token** — a second credential carried in an `X-Run402-Write-Auth` header that the gateway now requires *in addition to* the control-plane session bearer before a **wallet-less human operator** may provision or deploy. The dual-header gate (`requireWriteAuthCovers`) protects `provision` (`org.project.create`) and `deploy`/`apply` (`project.deploy`): a control-plane session alone "stays read-only on these routes" and returns `403 WRITE_AUTH_REQUIRED`.

The public CLI mints the control-plane **session** (`run402 operator login --loopback`) but has **zero write-auth-token plumbing** — no ceremony to mint it, no cache, no header. Worse, the default `NodeCredentialsProvider.getAuth` only ever returns wallet (SIWX) headers; it never falls back to the cached control-plane session. So today a wallet-less human can `operator login` and read their overview, but **every write fails**: first because the session is never injected into `provision`/`deploy`, then — once it is — because the write-auth token is absent. This change closes the wallet-less write path. (The SIWX wallet path that agents use is unaffected — `walletAuthOrWriteAuthProvision` still accepts wallet auth with no write-auth token.)

> **Scope note:** this is the **large** item from the upgrade review, not a small one. The discovery that `getAuth` has no control-plane fallback means the change spans `core` (token cache), the SDK kernel-adjacent credential path (dual-header injection), the operator ceremony, and the CLI — not just "wrap one endpoint." `design.md` records the option to split it (cp-session injection vs. write-auth token) if a smaller first slice is preferred.

## What Changes

- **NEW — write-auth token ceremony (isomorphic SDK seams).** `r.operator.writeAuth.requestChallenge({ cliRedirectUri, codeChallenge, state })` → `POST /agent/v1/control-plane/write-auth/challenges` (carries the control-plane session bearer; returns `{ confirm_url, delivery }`), and `r.operator.writeAuth.exchangeClaimCode({ code, codeVerifier, state })` → `POST /agent/v1/control-plane/write-auth/cli/token` → `{ write_auth_token, token_type, header, expires_in, … }`. Mirrors the existing `buildCliAuthorizeUrl` / `exchangeCliToken` session seams.
- **NEW — write-auth token cache (`core/`).** `core/src/write-auth-session.ts` mirrors `control-plane-session.ts`: atomic 0600 write of `{ write_auth_token, header, expires_at, principal_id?, amr? }` at `{base}/write-auth-session.json`, with `read/save/clear/isExpired/loadLive/fromTokenResponse`. Short-lived (gateway 30-min idle / 4-hr absolute); the cache stores the absolute `expires_at`.
- **NEW — dual-header injection via `getAuth` (no kernel change).** `NodeCredentialsProvider.getAuth(path)` gains a control-plane fallback: when no wallet allowance is present, it returns the live control-plane session as `Authorization: Bearer …` and, for the write paths, adds `X-Run402-Write-Auth: Bearer …` from the write-auth cache. The kernel already merges provider headers without overwriting, so no kernel edit is required.
- **NEW — CLI ceremony + wallet-less writes.** `run402 operator write-auth` runs the passkey write-intent dance headlessly: generate PKCE + state, start the 127.0.0.1 loopback server (reusing the existing `generatePkce` / loopback-server helpers), `requestChallenge`, open the `confirm_url`, capture `code` on the loopback redirect, `exchangeClaimCode`, cache the token. After it, `run402 provision` / `run402 deploy apply` work wallet-less.
- **NEW — typed `WriteAuthRequiredError`.** Map `403 WRITE_AUTH_REQUIRED` to a recognizable SDK error whose message points at `run402 operator write-auth`, so a wallet-less write lacking (or with an expired) token fails with guidance, not an opaque 403.
- **DOC FIX — control-plane session is no longer SIWX-equivalent for writes.** The `core/control-plane-session.ts` header comment ("write-capable … accepted everywhere a SIWX wallet is") is now stale for `provision`/`deploy`; correct it and document the dual-credential model + `WRITE_AUTH_*` codes in `cli/llms-cli.txt` / `sdk/llms-sdk.txt`.

## Impact

- **core (`core/src/`):** new `write-auth-session.ts` (cache; mirrors `control-plane-session.ts` byte-for-byte in shape and self-heal/atomic-write discipline).
- **SDK (`sdk/src/`):** `namespaces/operator.ts` — new `WriteAuth` sub-class (`requestChallenge`, `exchangeClaimCode`) + types; `node/credentials.ts` — control-plane + write-auth fallback in `getAuth`; `errors.ts` — `WriteAuthRequiredError` + kernel mapping for `403 WRITE_AUTH_REQUIRED`; export the new public types from the type-surface entry.
- **CLI (`cli/lib/`):** `operator.mjs` — `write-auth` verb (reuse `generatePkce` + the loopback server), cache the token, `WRITE_AUTH_REQUIRED` guidance; confirm `provision`/`deploy` surface the typed error and the dual-header rides automatically.
- **MCP:** none — write-auth is a browser + loopback ceremony, not an MCP tool. Document that MCP hosts establish write-auth via the CLI, then MCP writes inherit the cached credential.
- **Tests:** SDK unit (ceremony seams; `getAuth` returns the dual header on write paths and cp-session-only on read paths; `403 WRITE_AUTH_REQUIRED` → `WriteAuthRequiredError`); core unit (cache round-trip + expiry, mirroring `control-plane-session.test`); CLI e2e (ceremony happy path against a stubbed gateway + loopback; guidance on missing/expired token) registered in the `package.json` test allow-list; `sync.test.ts` SURFACE if an operator verb is added.
- **Docs:** `cli/llms-cli.txt`, `sdk/llms-sdk.txt` — the dual-credential wallet-less write path + `WRITE_AUTH_*` codes; fix the stale `control-plane-session.ts` comment.
- **Cross-repo:** gateway shipped (`016b7f8c`, `2b2055f4`, run402-private). **Live-gateway verification required before building** — confirm the exact `challenges` / `confirm` / `cli/token` request+response shapes and that the SIWX wallet path stays write-capable without a write-auth token.
