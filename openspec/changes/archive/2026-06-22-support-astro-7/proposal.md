## Why

Astro 7 is the current latest major, and a local compatibility trial showed `@run402/astro` tests and package build passing against `astro@7.0.0`. The published package still declares `astro >=6 <7`, while `run402 init astro` scaffolds even older dependencies, so new and upgrading users get avoidable peer warnings or stale starter apps.

## What Changes

- Widen `@run402/astro`'s supported Astro peer range to include Astro 7 while retaining Astro 6 compatibility.
- Bump the `@run402/astro` dev dependency and lockfile resolution to Astro 7 so CI exercises the latest supported major.
- Update the Astro scaffold generated by `run402 init astro` to current `@run402/astro`, `@run402/functions`, and Astro dependency ranges.
- Update `@run402/astro` docs and changelog language that currently says Astro 6-only.
- Add/adjust test coverage so peer-range, scaffold dependency, package build, and Astro integration behavior remain pinned.
- Do not adopt Astro 7-only route caching, `src/fetch.ts`, or Markdown pipeline behavior in this change; those remain separate product/runtime decisions.

## Capabilities

### New Capabilities

- `astro-version-compatibility`: Defines the supported Astro major-version contract for `@run402/astro`, including package peer ranges, latest-major CI coverage, and starter scaffold dependency freshness.

### Modified Capabilities

- None.

## Impact

- `astro/package.json`, root `package-lock.json`, and any generated package metadata for the independent `@run402/astro` release.
- `cli/lib/init-astro.mjs` scaffold dependency template and CLI tests that inspect generated project files.
- `astro/README.md` and `astro/CHANGELOG.md`; optionally `AGENTS.md` if the supported-Astro-major statement changes there.
- Astro package tests plus a build smoke path that runs after `core` and `sdk` build artifacts exist.
