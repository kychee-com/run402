# IMPORTANT: zkprover — Three Viable Paths Forward

**Date:** 2026-04-16
**Author:** Barry Volinskey + Claude (research + measurement)
**Status:** Active decision document — read this before starting any prover work
**Supersedes:** zkprover v0.1.0 winner decision (DD-12, Candidate D). That decision stands as a fallback; this document expands the option set.

---

## Why this document exists

The zkprover v0.1.0 evaluation (April 2026) selected Candidate D (RISC Zero) as the winner at $0.033/sig. A follow-up evaluation (v2.0, same month) tested 4 audited alternatives (E/F/G/H). Through that research we discovered:

1. **Candidate E (Circom + rapidsnark)** proved in 13.49 seconds at $0.011/sig — cheaper than D AND with 5 circuit audits vs D's 0.
2. **BUT** rapidsnark only supports Groth16, which requires a per-circuit Phase 2 ceremony. Our trust anchor rule (DD-34a) says: "math or pre-existing global ceremony — never a Kychee-run Phase 2."
3. **The key insight:** PLONK doesn't need a Phase 2 ceremony. And the prover is OUTSIDE the trust boundary — it can't forge proofs. So a custom unaudited Rust PLONK prover + the audited Circom circuit = no ceremony, 5 audits, fast native speed.

This document records the three viable paths so we never repeat this research.

## Decision criteria (permanent, from DD-v2.2 / DD-34a)

| Priority | Criterion |
|---|---|
| **1** | **Cost per signing ≤ $0.15** hard cap (compute + gas, all-in). Non-negotiable. |

## Path A — Custom native PLONK prover for Circom R1CS (LEADING)

### What

Write a Rust PLONK prover that consumes the existing audited `kysigned-approval.circom` R1CS. Same circuit that was already proven to work with rapidsnark (13.49s, 255k gas, $0.011/sig) — just a different proof system that doesn't need a Phase 2 ceremony.

### Trust chain

```
kysigned-approval.circom (source)     ← AUDITED by 5 firms
        ↓
circom compiler → R1CS constraints   ← Battle-tested (Polygon zkEVM, billions TVL)
        ↓
Custom Rust PLONK prover → proof     ← OUTSIDE trust boundary (can't forge, only fail)
        ↓
snarkjs Solidity PLONK verifier      ← Deterministic from SRS
        ↓
Hermez ppot universal SRS            ← 86+ contributors (Vitalik, EF, Zcash)
                                        NO Phase 2 ceremony needed
```

**Why the prover doesn't need auditing:** A ZK prover can only produce (a) a valid proof → verifier accepts (correct), (b) an invalid proof → verifier rejects (safe), or (c) crash (safe). It CANNOT produce a proof that a correct verifier accepts for a false statement. That would require breaking the discrete logarithm assumption, not a software bug. The security lives in the circuit + verifier + SRS, not the prover.

- **Pending:** Security consult to validate the full trust chain before committing effort.

## Path B — RISC Zero (Candidate D) with cfdkim hardening

### What

Current working solution. RISC Zero zkVM runs a Rust guest program that uses `cfdkim` for DKIM verification. STARK inner proof (math-only) + Groth16 wrapper (238-contributor PSE ceremony).

### Trust chain

```
cfdkim (Rust DKIM library)            ← UNAUDITED (0 firms). This is the risk.
        ↓
RISC Zero guest program               ← Faithfully executes cfdkim. Can't catch app bugs.
        ↓
RISC Zero STARK proof                  ← Math-only (no ceremony)
        ↓
RISC Zero Groth16 wrapper              ← 238-contributor PSE/EF ceremony (pre-existing, not ours)
        ↓
RiscZeroGroth16Verifier (Solidity)     ← Audited (Hexens, Veridise, Picus FV 45k+ constraints)
```

### Measurements (actual, from v0.1.0 evaluation)

| Metric | Value |
|---|---|
| Proving time | 3m 03s (STARK only, Groth16 wrap adds ~30-60s) |
| Peak RAM | 8.4 GB |
| On-chain gas | ~280k (documented, Groth16 wrapper) |
| **All-in $/sig** | **~$0.033** |
| Passes $0.15 cap? | **YES** (78% under) |

### The cfdkim audit gap

**This is the critical weakness.** `cfdkim` is a Rust DKIM verification library (fork of Cloudflare's). It has **zero audits in the ZK context**. If a bug is ever published as a CVE, opposing counsel in a legal proceeding could argue all historical signatures are unreliable.

**Mitigations in progress:**

1. **cfdkim patches (COMPLETE):** 5 soundness bugs fixed in `C:\Workspace-Kychee\cfdkim` (branch `soundness-fixes`). 48 tests passing. PR + outreach email ready for upstream submission to `zkemail/cfdkim`.
   - **Spec:** `cfdkim/docs/products/cfdkim-patches/cfdkim-patches-spec.md`
   - **Plan:** `cfdkim/docs/plans/cfdkim-patches-plan.md`
   - **PR description:** `cfdkim/docs/products/cfdkim-patches/pr-description.md`
   - **Outreach:** `cfdkim/docs/products/cfdkim-patches/outreach-email.md`
   - Fixes: F1 (b= removal), F2 (a= algorithm binding), F3 (simple header canonicalization), F4 (duplicate DKIM tags), F5 (RFC parsing tightening)

2. **Consult findings:** `run402/docs/consultations/cfdkim-zk-soundness-review.md`
   - 4 HIGH: #1 b= global replace, #2 a=↔key-type mismatch, #3 LF-only body hash, #5 duplicate Subject/To
   - 3 MEDIUM: #4 simple header canonicalization, #6 duplicate DKIM tags, #7 selector policy
   - 1 LOW: #8 RFC parsing permissiveness
   - 1 ARCH: verify_email_with_key() can't prove full RFC DKIM

3. **Guest-level guards (2R.B.G8):** Provisional assertion guards inside the RISC Zero guest program that reject known-problematic inputs even if cfdkim doesn't catch them. Defined in `run402/docs/plans/kysigned-plan.md` Phase 2R.B.G8. Guards include: reject duplicate headers, reject l= tag, reject LF-only line endings, reject ambiguous Subject/To.

4. **Formal verification option:** Aeneas FV ($40-80K, 2-3 months) could mathematically prove cfdkim implements RFC 6376 correctly for ALL inputs. Stronger than an audit. Vendors: Formal Land (contact@formal.land), Cryspen (cryspen.com).
   - Reference: `memory/reference_aeneas_formal_verification.md`

### Ceremony

RISC Zero's 238-contributor Groth16 wrapper ceremony. PSE/EF-coordinated. Blog post + public verification. Named participants. This meets our trust bar ("pre-existing global ceremony").

### Status

**Working today.** 2R.B.A0 swap to zkemail/cfdkim completed. Patches implemented. Can ship with guest guards as interim protection while pursuing formal verification.

## Path C — Reuse zk-email email_auth ceremony-backed circuits

### What

Use existing `emailauthlegacy` and/or `emailauthwithrecipient` Circom circuits that already have Phase 2 ceremony-backed Groth16 zkeys. Instead of a custom circuit, restructure the signing protocol to fit within these circuits' interface.

### Protocol change

**Current flow:** Signer replies "I APPROVE" in email body → custom circuit checks body content.

**New flow (two-proof double intent):**
1. **emailauthlegacy proof:** Signer adds "I APPROVE" to the email subject → circuit extracts subject as "command" → on-chain contract parses approval + doc binding from command.
2. **emailauthwithrecipient proof:** Signer forwards to `i_approve@kysigned.com` → circuit's recipient commitment proves the target address → on-chain contract verifies commitment matches `i_approve@kysigned.com`.

Both proofs verify the same email's DKIM signature. Double intent: explicit text ("I APPROVE") + explicit action (forwarding to approval address).

## Multi-proof defense-in-depth

**Key insight: all 3 paths can run in parallel.** A single signature submission can include proofs from 2 or 3 independent systems. If any one system has a bug published later, the others still stand.

| Combination | $/sig | Independent trust anchors |
|---|---|---|
| Path A + Path B | ~$0.017 | PLONK (no ceremony) + RISC Zero (238-contributor ceremony) |
| Path A + Path C | ~$0.026 | PLONK + email_auth ceremony (59-67 contributors) |
| Path B + Path C | ~$0.049 | RISC Zero + email_auth |
| All three | ~$0.055 | Maximum defense-in-depth |

**On-chain architecture:** Single transaction calls multiple verifiers, stores one signature record. The contract requires ALL proofs to pass. If any fails, the transaction reverts.

```solidity
// Dual-proof: both must verify for signature to be accepted
require(plonkVerifier.verify(plonkProof), "PLONK failed");
require(riscZeroVerifier.verify(riscZeroSeal, imageId, journal), "RISC Zero failed");
signatures[signatureId] = true;  // One record, two proofs
```

## Recommendation

**Build Path A (PLONK prover), keep Path B (RISC Zero) as fallback, validate Path C (email_auth) in parallel.**

1. **Immediate:** Run /consult on PLONK trust chain. If no gaps found, start the 2-4 week Rust build.
2. **Parallel:** Keep RISC Zero (Path B) as the working production path while PLONK prover is built. Ship cfdkim patches upstream + deploy guest guards (2R.B.G8).
3. **Parallel:** Download email_auth ceremony zkeys and validate with rapidsnark. If they work, this becomes an additional proof layer.
4. **Target:** Dual-proof (Path A + Path B) at ~$0.017/sig. Best of both worlds — audited circuit + no ceremony + working fallback.
