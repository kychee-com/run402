# R.3 — On-chain Verifier Strategy

**Date:** 2026-04-10
**Phase:** R (zk-email & KDF Research Spike)
**Status:** Complete

## Recommendation: Groth16

**Groth16 is the clear winner for kysigned's on-chain verification.**

## Comparison

| Criteria | Groth16 | PLONK | STARKs |
|----------|---------|-------|--------|
| **Proof size** | ~260 bytes | ~500-800 bytes | ~50-200 KB |
| **On-chain verification gas** | ~250K gas | ~350-500K gas | ~1-5M gas |
| **On-chain verification cost (Base L2)** | ~$0.01-0.05 | ~$0.02-0.10 | ~$0.50-2.00 |
| **Server-side proof generation** | <5s (128-core) | ~10-30s | ~30-60s |
| **Cost per proof (server)** | <$0.01 | ~$0.01-0.05 | ~$0.05-0.20 |
| **Trusted setup required** | Yes (one-time) | Universal (shared) | No |
| **Quantum resistance** | No | No | Yes |
| **Tooling maturity** | Excellent (snarkjs, rapidsnark) | Good (circom-compat) | Developing |
| **zk-email support** | Native (primary) | Experimental | Not integrated |

## Why Groth16

1. **Smallest on-chain footprint:** ~260 bytes + ~250K gas. At kysigned's $0.39/envelope price, on-chain verification is <$0.05 on Base — leaving healthy margin for gas + email + compute costs.

2. **Fastest server-side generation:** <5 seconds on a 128-core machine. The zk-email team runs autoscaled spot instances for <$0.01/proof. kysigned can either use the zk-email registry's proving infrastructure or self-host.

3. **Native zk-email integration:** The zk-email SDK uses Groth16 as its primary proof system. The `@zk-email/contracts` package includes a Groth16 verifier template generated by snarkjs. No custom integration work needed.

4. **Audited:** The zk-email Groth16 circuits have been audited by zkSecurity and yAcademy (v6.1.0).

## Trusted Setup

Groth16 requires a trusted setup ceremony per circuit. Options:

1. **Use zk-email's existing Powers of Tau:** The zk-email project has completed ceremonies for their standard circuits. If our kysigned Blueprint extends the standard EmailVerifier, we can reuse their Phase 1 (universal) setup and only need a Phase 2 (circuit-specific) contribution.

2. **Run our own Phase 2:** After the circuit is compiled, run `snarkjs groth16 setup` with the zk-email Phase 1 `.ptau` file. Contribute randomness, generate the proving key and verification key. One-time cost.

3. **Registry handles it:** When submitting a Blueprint via the zk-email SDK registry, the compilation process includes setup. The registry returns a compiled circuit with proving/verification keys.

**Recommendation:** Use the SDK registry's compilation (option 3) for MVP. If self-hosting proving becomes necessary, run our own Phase 2 (option 2).

## Quantum Resistance Note (DD-20)

Groth16 relies on elliptic curve pairings vulnerable to Shor's algorithm. This is acknowledged in DD-20: "document, don't mitigate." The upgrade path is STARKs when the tooling matures and gas costs drop. The `EvidenceKeyRegistry`'s blockchain-timestamped key archival makes historical verification meaningful even after a future cryptographic break.

## Verifier Contract

The Groth16 verifier contract is auto-generated by `snarkjs` from the circuit's verification key:

```bash
snarkjs zkey export solidityverifier circuit_final.zkey KysignedVerifier.sol
```

This produces a single-function contract: `verifyProof(uint[2] a, uint[2][2] b, uint[2] c, uint[N] input) → bool`. Deployed once, called by `SignatureRegistry.recordReplyToSignSignature()` to validate the proof before recording.

## Decision

**Use Groth16 via the zk-email SDK Blueprint system.** Deploy the auto-generated `KysignedVerifier.sol` alongside the rewritten `SignatureRegistry.sol` in Phase 1R. No PLONK or STARK work needed for MVP.
